# 视频画面内容检索平台项目总结

## 1. 项目定位

这是一个面向屏幕录制视频审查的检索平台，用于把无声录屏视频转成可检索证据，帮助定位以下风险场景：

- 使用 AI 对话平台或 AI 编码助手
- 在考试、编程或审查场景中疑似借助 AI 完成任务
- 打开预先准备好的答案、代码、文档或操作步骤
- 通过关键词和全文搜索快速定位可疑时间段

项目目标不是生成完整视频理解报告，而是尽快把视频拆成结构化帧数据，支持“检索 + 回看 + 定位时间段”。

---

## 2. 当前技术方案

### 后端

- 框架：FastAPI
- 数据库：SQLite + FTS5
- 视频处理：FFmpeg / ffprobe
- 任务队列：内存异步队列
- 模型接入：OpenAI-compatible API
- 当前默认视觉模型：`glm-4.6v-flash`

### 前端

- 框架：React 19 + Vite
- 路由：React Router
- 样式：Tailwind CSS + 自定义样式

---

## 3. 主要功能

当前项目已具备以下能力：

- 导入单个视频
- 批量导入文件夹
- 自动探测视频元信息
- 按固定时间间隔抽帧
- 将每帧发送给视觉模型做结构化分析
- 将分析结果写入 SQLite
- 用 FTS5 做全文搜索
- 中文搜索分词支持
- 搜索结果聚合为“可疑时间段”
- 关键词集合管理与批量扫描
- 导入进度展示
- 结果详情页查看帧图、分析结果和对应视频时间点
- Docker Compose 部署

---

## 4. 当前默认配置

项目当前默认配置偏向“控制成本 + 提高稳定性”：

- 模型：`glm-4.6v-flash`
- 模型并发：`API_CONCURRENCY=1`
- 抽帧间隔：`FRAME_INTERVAL=5`
- 抽帧宽度：`FRAME_MAX_WIDTH=1280`
- JPEG 质量：`FRAME_JPEG_QUALITY=5`

这套配置的目的：

- 降低单帧图片体积
- 减少总帧数
- 降低总 token 消耗
- 避免接口 429 限流

---

## 5. 处理流程

当前视频处理链路如下：

1. 用户导入视频或文件夹
2. 后端校验路径并用 `ffprobe` 读取视频元数据
3. 创建 `video_assets` 和 `processing_tasks` 记录
4. FFmpeg 按设定间隔抽帧并压缩成 JPEG
5. 每帧调用视觉模型获取结构化 JSON
6. 将分析结果写入 `frame_analysis`
7. 将扁平化文本写入 `frame_analysis_fts`
8. 前端可按关键词检索帧结果和聚合时间段

### 搜索能力

- 修复中文全文搜索不可用问题，接入 `jieba`
- 修复多词搜索逻辑，查询词改为 `AND`，减少误命中
- 新增可疑时间段聚合展示

- 单视频导入前校验文件路径是否存在且为文件
- 文件夹导入前校验目录是否存在且为目录
- `ffprobe` 异常转换为 400 级错误，避免 500 泄漏

### 并发与进度

- 真正启用配置化并发队列
- 修复“假并发”实现
- 修复导入进度条长时间不动的问题
- 任务进度现在按帧完成情况持续更新

### 模型稳定性

- 默认模型切换为更便宜的 `glm-4.6v-flash`
- 对 429 限流增加更长的退避重试
- 修复模型偶发返回脏 JSON 导致整段任务失败的问题
- 解析器现在可容忍控制字符、尾逗号、缺失逗号等常见瑕疵

### 前端稳定性

- 修复搜索页因旧接口返回结构不完整而崩溃的问题
- 增加路由级错误边界
- 搜索接口返回做前端归一化，兼容旧响应格式

### Docker 化

- 修复前端部署时 API 地址默认指向本地 `127.0.0.1` 的问题
- 前端默认改为走同源 `/api`
- 更新 `docker-compose.yml`，适配当前环境变量和数据卷

## 8. 数据存储说明

### 数据库

- 默认路径：`data/db/search.db`

### 帧文件

- 默认目录：`data/frames`
- 每个视频对应一个目录：
  - `data/frames/video_0001/`
  - `data/frames/video_0002/`

---

## 9. 部署方式

当前推荐使用 Docker Compose 部署：

```bash
docker compose up --build -d
```

部署后：

- 前端默认端口：`80`
- 后端默认端口：`8000`

如果在 Docker 中导入视频，需要在前端输入容器内路径，例如：

```text
/app/videos/使用AI平台.mp4
```

---

## 10. 当前已知限制

- 任务队列是内存队列，服务重启后不会自动恢复未完成任务
- 仍然是按帧调用远程视觉模型，长视频处理速度受模型接口吞吐限制明显
- 当前没有做“粗扫 + 精扫”的两阶段处理
- SQLite 适合 MVP 和单机部署，不适合高并发多用户生产场景
- 当前 CORS 仍较宽松，生产环境应收紧

---

## 11. 后续建议

如果继续迭代，优先级建议如下：

1. 增加“快速扫描 / 标准扫描”模式
2. 做两阶段分析：先粗筛可疑时间段，再对命中片段精细复扫
3. 增加任务恢复和失败重试策略
4. 增加结果导出能力
5. 如果要上多用户或更大规模视频，考虑把任务队列和数据库替换为更正式的基础设施

---

## 12. 当前结论

这个项目已经不再只是一个原型壳：

- 可以真实导入视频
- 可以完成抽帧、模型分析、落库、检索和结果回看
- 已支持中文搜索
- 已支持时间段聚合
- 已支持 Docker 部署
- 已针对成本、限流、脏 JSON、前端崩溃和导入进度问题做过一轮实战修复

如果作为单机内部工具或小规模审查工具使用，当前版本已经可用。
