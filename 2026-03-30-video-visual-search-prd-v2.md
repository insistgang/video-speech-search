## 1. 背景与问题定义

### 1.1 业务背景

当前需要对大量视频进行快速检索与排查。目标视频为**屏幕录制类视频**，记录了操作者在电脑上的操作过程，**没有语音**，核心信息全部体现在画面内容中。

业务上关心的核心问题是：

- 指定操作行为是否在某批视频中出现过（如使用 AI 平台、调用 API 等）
- 是否存在使用特定工具或平台的行为（如 AWS Bedrock、CLI 工具、IDE 插件等）
- 是否存在提前准备好答案并直接使用的行为
- 是否可以针对一组重点关键词和行为模式进行批量排查
- 是否能在命中后快速定位到对应时间点完成复核

### 1.2 视频特征

根据实际样本分析，目标视频具有以下特征：

- **无语音**：视频中没有任何语音内容，不适用 ASR 方案
- **纯屏幕录制**：画面内容为操作者的电脑屏幕
- **包含大量文字信息**：网页标题、代码内容、AI 对话界面、文件名、菜单项等
- **操作行为可识别**：打开浏览器、切换标签页、复制粘贴、运行命令等

已知视频样本分类：

| 视频文件名 | 所属目录 | 内容特征 |
|---|---|---|
| 使用AI平台.mp4 | 作弊视频/AI的使用 | 使用 AI 对话平台 |
| 使用AWS平台内部AI.mp4 | 作弊视频/AI的使用 | 使用 AWS 内置 AI 服务 |
| 使用CLI调用Bedrock生成代码文件.mp4 | 作弊视频/AI的使用 | 通过命令行调用 AWS Bedrock |
| 在IDE侧使用AI.mp4 | 作弊视频/AI的使用 | 在 IDE 中使用 AI 辅助编码 |
| 使用提前准备好的步骤.mp4 | 作弊视频/使用提前准备好的内容 | 使用预置操作步骤 |
| 使用提前准备好的代码.mp4 | 作弊视频/使用提前准备好的内容 | 使用预置代码文件 |

### 1.3 核心问题

当前处理方式存在以下问题：

- 视频数量多，人工逐条观看效率极低
- 无语音视频无法通过传统 ASR 方案处理
- 画面中包含大量文字和操作信息，但缺乏自动化提取手段
- 仅依赖人工经验，难以形成稳定排查流程
- 难以对重点行为和关键词进行规模化筛查和留痕

### 1.4 解决思路

本产品主路径确认为：

1. 从视频中提取关键帧（按固定间隔或场景变化）
2. 对关键帧调用多模态大模型（Kimi K2.5）进行画面理解
3. 生成带时间戳的结构化文本描述（画面内容、文字识别、操作行为）
4. 建立检索索引
5. 支持关键词检索、模糊检索和语义检索
6. 返回命中视频、文本片段和时间点供人工复核

**备选增强路径**：对视频直接通过 Kimi K2.5 video_url 接口整段理解（适用于短视频），或对长视频切片后逐段理解。

## 2. 产品目标与成功指标

### 2.1 产品目标

- 支持从无语音屏幕录制视频中提取画面内容并转化为可检索文本

- 支持返回命中视频、命中时间点和相关文本片段
- 支持基于已知关键词库进行定向筛查
- 为后续扩展（操作行为识别、自动风险判定）预留能力

| 已知关键词召回率 | >= 80% | 基于测试样本集 |
| 人工复核耗时下降 | >= 70% | 相比全量人工观看 |

## 3. 用户角色与典型场景

### 3.1 用户角色

- **审核人员**：排查视频中是否出现指定操作行为或内容
- **分析人员**：通过关键词或意图描述搜索相关视频片段
- **管理人员**：维护关键词库、风险行为库和检索规则

### 3.2 典型场景

- 用户导入一个包含 100 个视频的文件夹，输入"使用 AI"，系统返回画面中出现 AI 工具使用行为的视频及对应时间点
- 用户输入"有没有人打开了 ChatGPT"，系统返回画面中出现类似 AI 对话界面的视频片段
- 用户输入"Bedrock"，系统返回画面中出现 AWS Bedrock 控制台或 CLI 调用的视频
- 用户提前维护一批重点关键词，例如"AI 助手""代码生成""提前准备""标准答案"，系统支持快速筛查
- 用户查看命中结果时，可直接看到命中的画面截图和对应时间段，便于复核
- 后续可支持输入视频链接，自动抓取并纳入检索

## 5. 业务流程

### 5.1 主流程

```text
视频导入
  -> 关键帧提取（ffmpeg 按间隔/场景变化抽帧）
  -> [路径A] 关键帧逐帧调用 Kimi K2.5 image 接口理解画面
  -> [路径B] 视频切片后调用 Kimi K2.5 video_url 接口整段理解
  -> 生成带时间戳的结构化描述文本
  -> 文本清洗与切分
  -> 建立检索索引
  -> 用户输入关键词/自然语言
  -> 返回命中视频与时间点
  -> 人工复核（展示命中截图 + 视频跳转）
```

### 5.2 路径选择策略

| 条件 | 推荐路径 | 说明 |
|---|---|---|
| 视频时长 < 60 秒 | 路径 B（整段视频理解） | Kimi K2.5 video_url 接口，信息更完整 |
| 视频时长 60-300 秒 | 路径 B（切片后视频理解） | 按 60 秒切片，逐段调用 |
| 视频时长 > 300 秒 | 路径 A（关键帧理解） | 按 2-5 秒间隔抽帧，逐帧调用 |
| 需要精确 OCR 文字 | 路径 A（关键帧理解） | 图片模式 OCR 识别更精确 |

### 5.3 异常流程

- 视频格式不支持：任务失败并记录原因
- 关键帧提取失败：允许重试或人工跳过
- 多模态 API 调用失败：记录错误，允许重试或切换模型
- API 限流：实现队列机制和退避重试
- 检索无结果：提示无命中，并支持放宽检索条件

## 6. 技术方案

### 6.1 多模态 API 选型

主选方案：**Kimi K2.5**（Moonshot AI）

选型理由：

- 原生多模态架构，视频理解能力突出（VideoMMU 86.6%）
- OCR 识别能力强（OCRBench 92.3%），适合屏幕录制文字识别
- 中文理解能力一流，适合中文界面识别
- 兼容 OpenAI API 格式，接入成本低
- 价格优势明显（$0.60/$2.50 per million tokens）
- 支持 base64 编码的图片和视频输入

### 6.2 Kimi K2.5 API 调用方式

**图片理解（关键帧模式）**：

```python
from openai import OpenAI
import base64

client = OpenAI(
    api_key=os.environ.get("MOONSHOT_API_KEY"),
    base_url="https://api.moonshot.cn/v1",  # 国内端点
)

with open("frame_001.png", "rb") as f:
    image_base64 = base64.b64encode(f.read()).decode("utf-8")

completion = client.chat.completions.create(
    model="kimi-k2.5",
    messages=[{
        "role": "user",
        "content": [
            {"type": "image_url", "image_url": {"url": f"data:image/png;base64,{image_base64}"}},
            {"type": "text", "text": "请详细描述这个屏幕截图的内容..."}
        ]
    }]
)
```

**视频理解（整段/切片模式）**：

```python
with open("clip_001.mp4", "rb") as f:
    video_base64 = base64.b64encode(f.read()).decode("utf-8")

completion = client.chat.completions.create(
    model="kimi-k2.5",
    messages=[{
        "role": "user",
        "content": [
            {"type": "video_url", "video_url": {"url": f"data:video/mp4;base64,{video_base64}"}},
            {"type": "text", "text": "请详细描述这个屏幕录制视频的内容..."}
        ]
    }]
)
```

### 6.3 关键帧提取方案

使用 FFmpeg 提取关键帧：

```bash
# 按固定间隔（每 3 秒一帧）
ffmpeg -i input.mp4 -vf "fps=1/3" -q:v 2 frames/frame_%04d.jpg

# 按场景变化检测
ffmpeg -i input.mp4 -vf "select='gt(scene,0.3)'" -vsync vfr frames/frame_%04d.jpg
```

### 6.4 Prompt 设计

对每帧/每段视频的理解 Prompt 需要结构化输出：

```
你是一个屏幕录制视频内容分析专家。请分析这个屏幕截图/视频片段，返回以下 JSON 格式的结构化信息：

{
  "screen_text": "画面中可见的所有文字内容",
  "application": "当前使用的应用程序或网站名称",
  "url": "如果可见，当前浏览器地址栏的 URL",
  "operation": "用户正在执行的操作描述",
  "ai_tool_detected": true/false,
  "ai_tool_name": "如检测到 AI 工具，给出工具名称",
  "code_visible": true/false,
  "code_content": "如可见代码，给出代码片段摘要",
  "risk_indicators": ["检测到的可疑行为列表"],
  "summary": "一句话总结画面内容"
}

重点关注以下行为：
1. 使用 AI 对话平台（ChatGPT、Claude、Kimi、文心一言等）
2. 使用云平台 AI 服务（AWS Bedrock、Azure OpenAI 等）
3. 通过 CLI/终端调用 AI API
4. 在 IDE 中使用 AI 代码助手（Copilot、Cursor 等）
5. 打开预先准备好的文件或答案
6. 从外部来源复制粘贴内容
```

## 8. 详细功能需求

### 8.1 视频接入

#### 8.1.1 P0 需求

- 支持导入单个本地视频文件
- 支持导入本地文件夹并批量处理多个视频
- 支持记录视频基础信息，包括文件名、路径、格式、时长、分辨率、导入时间
- 支持任务队列，展示处理中、成功、失败状态

#### 8.1.2 P2 需求

- 支持输入视频链接并自动抓取处理

### 8.2 关键帧提取

#### 8.2.1 P0 需求

- 使用 FFmpeg 从视频中按固定间隔（默认 3 秒）提取关键帧
- 每帧保存为图片文件，并记录对应的时间戳
- 支持配置抽帧间隔（1-10 秒可调）
- 保留帧图片与原视频的映射关系

#### 8.2.2 P1 需求

- 支持基于场景变化检测的智能抽帧（减少冗余帧）
- 支持视频切片（按时长分段），用于 video_url 接口

### 8.3 多模态画面理解

#### 8.3.1 P0 需求

- 对每个关键帧调用 Kimi K2.5 API 进行画面理解
- 理解结果必须包含结构化字段：画面文字、应用名称、操作描述、AI 工具检测、风险指标
- 理解结果以 JSON 格式存储，关联帧图片和时间戳
- 支持配置自定义 Prompt，适配不同排查场景
- 支持 API 调用失败时的自动重试和错误记录

#### 8.3.2 P1 需求

- 支持通过 video_url 接口对短视频或视频切片进行整段理解
- 支持多模型切换（如备选 Gemini、GPT-4o 等）

### 8.4 文本处理与索引

#### 8.4.1 P0 需求

- 将多模态理解的结构化结果展平为可检索文本
- 支持按帧、按时间窗口切分和聚合文本
- 支持保留原文与时间点、帧图片的对应关系
- 建立全文检索索引（基于 SQLite FTS 或 Elasticsearch）

#### 8.4.2 P1 需求

- 支持向量化存储，用于语义检索

### 8.5 检索能力

#### 8.5.1 P0 需求

- 支持关键词精确检索（在画面文字和描述中匹配）
- 支持短句检索
- 支持在指定视频范围、时间范围、任务批次内搜索
- 支持按相关度、命中次数、导入时间排序

#### 8.5.2 P1 需求

- 支持模糊匹配，识别近似表达
- 支持语义检索，通过向量或大模型理解用户意图

### 8.6 结果展示

#### 8.6.1 P0 需求

- 返回命中的视频列表
- 每条结果展示：视频名称、命中文本、命中时间点、命中帧截图
- 支持展示前后文片段和相邻帧截图
- 支持定位到视频对应时间段，便于快速复核
- 展示 AI 工具检测结果和风险指标

#### 8.6.2 P1 需求

- 支持导出结果（CSV/Excel），供后续分析或留档
- 支持批量标记命中结果的风险等级

### 8.7 关键词库管理

#### 8.7.1 P0 需求

- 支持维护重点关键词、短语和专题词库
- 支持按专题管理，例如"AI 工具使用""预置内容""网络搜索"
- 支持关键词库新增、编辑、删除
- 支持基于词库执行快速筛查

## 11. MVP 范围

### 11.1 MVP 必须包含

- 本地单视频和文件夹导入
- FFmpeg 关键帧提取（固定间隔）
- Kimi K2.5 图片理解 API 调用
- 结构化 JSON 结果存储
- 关键词精确检索
- 返回视频名、命中文本、命中时间点、命中帧截图
- 基础关键词库管理

### 11.2 MVP 暂不包含

- 视频链接自动抓取
- video_url 整段视频理解
- 高级语义检索（向量）
- 场景变化智能抽帧
- 风险自动判定报告

## 13. 非功能需求

### 13.1 性能

- 支持至少 100 个视频的批量处理
- 在已建立索引的前提下，检索响应时间应达到秒级
- API 调用需实现并发控制和队列管理，避免触发限流

### 13.2 成本控制

- 记录每个视频的 API Token 消耗
- 支持配置抽帧间隔以平衡成本与精度
- 提供成本预估功能（导入前预估）

### 13.3 可用性

- 保留任务日志、API 调用日志和检索日志，便于审计与追溯
- 支持失败重试和基础状态监控

### 13.4 可扩展性

- 多模态理解模块应具备可替换性（Kimi K2.5 / Gemini / GPT-4o）
- 检索引擎应支持从 SQLite FTS 升级到 Elasticsearch
- Prompt 模板应可配置，支持不同排查场景

### 13.5 安全与合规

- 支持私有化部署或本地部署，满足敏感内容处理要求
- 视频和截图数据需加密存储
- 检索结果需具备可解释性，能够展示命中依据
- 重要操作应保留留痕

## 14. 验收标准

- 用户可成功导入单个视频及视频文件夹
- 系统可自动完成关键帧提取和多模态理解
- 理解结果包含结构化字段（画面文字、应用名称、操作描述等）
- 用户输入关键词后，系统可返回命中视频列表
- 每条结果至少显示视频名、命中文本、命中时间点、命中帧截图
- 对已知 AI 工具使用行为，系统能够稳定检出
- 用户可通过关键词库对重点内容进行快速筛查

## 15. 依赖、风险与限制

### 15.1 依赖

- FFmpeg：关键帧提取
- Kimi K2.5 API：多模态画面理解
- OpenAI Python SDK：API 调用（兼容格式）
- 检索引擎：SQLite FTS5（MVP）/ Elasticsearch（后续）

### 15.2 风险与限制

- 视频画质差、分辨率低会影响画面理解准确率
- API 调用成本与视频数量和抽帧密度成正比，需要控制成本
- Kimi K2.5 API 可能存在限流，需要实现队列和退避策略
- 部分画面可能包含小字体文字，OCR 识别率可能下降
- 结构化输出依赖 Prompt 质量，需要持续优化
- 整段视频理解受限于 API 的视频大小限制（需确认 Kimi K2.5 的具体限制）

### 15.3 待确认事项

- Kimi K2.5 video_url 接口的视频大小上限
- API 并发调用的限流策略和配额
- 是否需要全离线部署（如需，则需评估本地部署的开源多模态模型）
- 是否需要保留原始关键帧图片作为证据材料
- 是否需要导出标准化报告
- 是否需要权限管理与操作审计
- 国内访问使用 `api.moonshot.cn` 还是 `api.moonshot.ai` 端点
